# Publishing to Google Play (local build)

Step-by-step for shipping a new `.aab` to the Play Store from your
Mac, without any cloud build service. First publish walks through
Play Console setup too; subsequent updates are just steps 4–6.

## 0. One-time: Play Console account

- Create at <https://play.google.com/console>, pay the $25 fee.
- Choose **Personal** account type unless you need the company name
  shown publicly (Organization requires a D-U-N-S number).
- Upload a government ID. Verification is usually 24–48 hours.

## 1. One-time: generate the upload keystore

This keystore signs every release. **Back it up somewhere durable —
if you lose it, you cannot publish updates to this app's listing.**
(Play App Signing adds a recovery path, but don't rely on it.)

Run on your Mac, outside the repo:

```bash
mkdir -p ~/keystores
keytool -genkeypair -v \
  -keystore ~/keystores/voice-checklist-upload.jks \
  -alias upload \
  -keyalg RSA -keysize 2048 -validity 10000
```

Pick a strong passphrase and **write it down in your password
manager**. Use the same passphrase for the keystore and the key
entry (keytool will ask twice). Answer the distinguished-name
questions honestly — they're embedded in the cert but Play doesn't
rely on them.

Confirm:

```bash
keytool -list -v -keystore ~/keystores/voice-checklist-upload.jks
```

Back up the `.jks` file — iCloud Drive, a password manager with
file attachments, or an encrypted disk image all work.

## 2. One-time: tell Gradle how to sign releases

Store the passphrase outside the repo — put it in the user-level
Gradle properties so it's shared across projects and never checked
in:

```bash
cat >> ~/.gradle/gradle.properties <<'EOF'

VOICE_CHECKLIST_UPLOAD_STORE_FILE=/Users/michaelb/keystores/voice-checklist-upload.jks
VOICE_CHECKLIST_UPLOAD_KEY_ALIAS=upload
VOICE_CHECKLIST_UPLOAD_STORE_PASSWORD=<your passphrase>
VOICE_CHECKLIST_UPLOAD_KEY_PASSWORD=<your passphrase>
EOF

chmod 600 ~/.gradle/gradle.properties
```

Then add a `release` signing config and wire it into the `release`
build type in `android/app/build.gradle`. Open that file and change:

```groovy
signingConfigs {
    debug {
        storeFile file('debug.keystore')
        storePassword 'android'
        keyAlias 'androiddebugkey'
        keyPassword 'android'
    }
}
```

to:

```groovy
signingConfigs {
    debug {
        storeFile file('debug.keystore')
        storePassword 'android'
        keyAlias 'androiddebugkey'
        keyPassword 'android'
    }
    release {
        if (project.hasProperty('VOICE_CHECKLIST_UPLOAD_STORE_FILE')) {
            storeFile file(VOICE_CHECKLIST_UPLOAD_STORE_FILE)
            storePassword VOICE_CHECKLIST_UPLOAD_STORE_PASSWORD
            keyAlias VOICE_CHECKLIST_UPLOAD_KEY_ALIAS
            keyPassword VOICE_CHECKLIST_UPLOAD_KEY_PASSWORD
        }
    }
}
```

And in the same file, change the `release` build type's `signingConfig`
from `signingConfigs.debug` (a placeholder that ships with Expo's
template) to `signingConfigs.release`:

```groovy
buildTypes {
    debug {
        signingConfig signingConfigs.debug
    }
    release {
        signingConfig signingConfigs.release
        // ... existing lines (shrinkResources, minifyEnabled, proguardFiles) untouched ...
    }
}
```

Note: `android/` is gitignored (regenerated by `expo prebuild`). If
you ever re-prebuild, you'll need to re-apply these edits. Worth
pinning them into a config plugin later if prebuild becomes routine.

## 3. Bump the version for each release

In `app.json`, before each release, bump:

```json
"expo": {
  "version": "1.0.1",       // user-visible semver
  "android": {
    "versionCode": 2,       // monotonically increasing integer; Play rejects duplicates
    ...
  }
}
```

`versionCode` must strictly increase with every upload. `version`
(the name users see) is free-form semver.

Then run:

```bash
npx expo prebuild --platform android --no-install
```

so `versionCode` / `versionName` propagate from `app.json` into
`android/app/build.gradle`. *(Re-apply the signing config from step 2
afterward if prebuild overwrote it.)*

## 4. Build the signed `.aab`

```bash
cd android
./gradlew bundleRelease
```

Output: `android/app/build/outputs/bundle/release/app-release.aab`
(≈ 40–60 MB).

Sanity-check the signature matches your upload key:

```bash
jarsigner -verify -verbose -certs \
  android/app/build/outputs/bundle/release/app-release.aab \
  | head -20
```

## 5. First publish: create the Play Console listing

Once your Play account is verified, in [Play Console](https://play.google.com/console):

1. **Create app**
   - App name: "Voice Checklist"
   - Default language: English (United States)
   - App or game: App
   - Free or paid: Free
   - Accept the declarations.

2. **Set up your app** (left-nav checklist). Fill these in any order:
   - **Main store listing** — paste from `docs/store-listing.md`; upload the 4 screenshots in `docs/play-store/screenshots/`; add a 512×512 icon (the one in `assets/images/icon.png` will do) and a 1024×500 feature graphic (you'll need to make this — a title card on solid background is fine).
   - **Privacy policy** (under App content) — URL of your hosted `PRIVACY.md`. Enable GitHub Pages on this repo (Settings → Pages → source = `main` / `/docs`) to get `https://mbuchoff.github.io/voice-driven-checklist/PRIVACY`.
   - **Ads** — No.
   - **App access** — All functionality available without special access.
   - **Content rating** — fill out the questionnaire; all answers are "no" for this app. Expect IARC 3+.
   - **Target audience** — age 13+.
   - **Data safety** — "no data collected or shared" (full guidance in `docs/store-listing.md`).
   - **Government apps** — No.
   - **News apps** — No.
   - **COVID-19 contact tracing** — No.

3. **Testing → Internal testing**
   - Create a release: upload `app-release.aab`.
   - Release name defaults to the versionName; release notes free-form ("Initial internal build").
   - Review and roll out to Internal testing.
   - Add testers: create a tester list with `mbuchoff@gmail.com` and any friends' emails you want.
   - Copy the opt-in URL and visit it on your Pixel signed into the tester email — you'll then see the app in Play Store and can install it from there.

4. **Promote when ready**: Internal → Closed → Open → Production.
   Production submission triggers Google's first-time review (~3–7
   days historically). Subsequent production updates review faster.

## 6. Subsequent updates

1. Bump `version` and `versionCode` in `app.json` (step 3).
2. `./gradlew bundleRelease` (step 4).
3. Play Console → Testing/Production → Create release → upload new
   `.aab` → release notes → roll out.

That's it. No services, no accounts beyond Play Console and your
Google identity.
